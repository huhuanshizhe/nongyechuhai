# farmetra（农业出海）产品说明文档

> 版本：v0.1 | 更新日期：2026-05-21

---

## 1. 产品概述

### 1.1 产品简介

**farmetra** 是一个跨境 B2B 农业贸易平台，连接经过筛选的中国优质农产品供应商与全球买家（进口商、分销商、零售采购团队、餐饮服务及机构买家）。

不同于杂乱的 B2B 市场列表，farmetra 是一个"精选供应项目"展示平台：每个产品线都附有品类逻辑、供应商背景、质资认证及商业路线框架。

### 1.2 核心价值主张

| 维度 | 说明 |
|------|------|
| **供应商认证制** | 供应商提交资质文件，经由管理员审核后才能上线，而非泛泛收录 |
| **商业路线透明** | 每件产品标注交易模式——询盘驱动（Inquiry Program）或直接采购（Direct Order Program） |
| **结构化询盘** | 询盘表单一次性捕获目标市场、数量、包装和认证需求，减少来回沟通 |
| **48小时首次响应** | 对买家询盘承诺首次响应时限 |

### 1.3 目标用户

| 角色 | 典型用户 | 入口 |
|------|---------|------|
| **ADMIN（管理员）** | 平台运营团队 | admin 后台 (`apps/admin`) |
| **SUPPLIER（供应商）** | 经过审核的中国农产品出口商 | supplier 工作台 (`apps/supplier`) |
| **BUYER（买家）** | 全球进口商、零售采购团队、餐饮分销商 | web 商店前端 (`apps/web`) |
| **GUEST（游客）** | 未登录的潜在买家 | web 商店前端，可浏览产品和提交询盘 |

## 2. 用户角色体系

### 2.1 ADMIN — 平台运营后台

- **入口**：独立的登录页面，凭据验证后进入后台管理界面
- **权限范围**：查看和治理平台所有数据
- **核心职责**：
  - 供应商入驻审核（资质审查、状态管理）
  - 商品治理（审核上下架、信息完整性）
  - 询盘流转管理（查看全平台买家的采购需求）
  - 内容管理（CMS 页面和 FAQ 资产的发布维护）
- **侧边栏导航**：仪表盘 → 供应商管理 → 商品管理 → 询盘管理 → 内容管理

### 2.2 SUPPLIER — 供应商工作台

- **入口**：独立登录页面，认证时需角色为 `SUPPLIER` 且已通过审核
- **数据隔离**：供应商只能看到自己组织的商品、询盘和订单
- **关联机制**：通过 `User → OrganizationMember → Organization → Supplier` 链路确定归属
- **核心职责**：
  - 管理自有商品信息
  - 查看和跟进分配给自己的买家询盘
  - 处理订单流程
- **侧边栏导航**：商品管理 → 询盘管理 → 订单管理

### 2.3 BUYER — 公共商店前端

- **入口**：面向全球买家的公共网站，支持英语和中文
- **已登录买家**：可浏览产品、提交流程化询盘、在个人工作台追踪询盘和订单历史
- **游客（未登录）**：可浏览全部产品目录、查看产品详情、提交询盘（无需注册）
- **买家注册**：当前阶段由管理员在后台创建账号，暂未开放自助注册

## 3. 应用模块详述

### 3.1 公共商店前端（apps/web）

#### 3.1.1 首页

- **路径**：`/`（自动重定向到 `/en` 或 `/zh`）
- **数据来源**：`getHomepageData()` + `getStorefrontShellData()`
- **内容区块**：
  - **Hero 区域**：精选产品展示 + 平台可信度信号（认证供应商数、产品品类数、在线产品数、48小时响应承诺）
  - **平台标准**：三条运营原则（审核制供应商、买家导向的资质呈现、跨境执行纪律）
  - **精选产品**：最多 3 个已发布产品的卡片展示
  - **运营模式说明**：平台如何将传统农产品采购转化为结构化跨境销售工作流
  - **贸易流程展示**：4 步买家旅程（查看品类与产地 → 浏览出口档案 → 提交结构化询盘 → 协调报价与执行）
  - **品类聚焦**：活跃产品品类列表
  - **CTA 区域**：引导买家发起询盘或浏览产品目录
- **缓存策略**：ISR，每 5 分钟重新验证

#### 3.1.2 产品目录页

- **路径**：`/products?category=X&mode=inquiry|direct`
- **数据来源**：`getCatalogPageData(filters)`
- **功能特性**：
  - **品类筛选**：左侧品类侧边栏，按 slug 过滤
  - **商业路线筛选**：全部 / 询盘项目 / 直采项目 三个筛选项
  - **产品分组**：蘑菇品类按产品形态分为 Fresh / Dried / Retail 三组展示
  - **供应商信息面板**：展示供应商名称、所在地、产品线数量、认证状态
  - **空状态处理**：无匹配产品时给出提示和重置筛选入口

#### 3.1.3 产品详情页

- **路径**：`/products/:slug`
- **数据来源**：`getProductDetail(slug)`
- **内容区块**：
  - **图片画廊**：主图 + 缩略图
  - **核心信息**：品类、交易模式徽章、产品名称、摘要
  - **供应商信息**：名称、产地、认证标识
  - **价格标签**：价格区间（直采模式）或"通过询盘获取报价"（询盘模式）
  - **规格亮点**：从 specs_json 提取前 4 项关键规格
  - **产品变体**：SKU、价格、库存参考
  - **富文本描述**：完整的产品介绍
  - **FAQ 区域**：关联到该产品的常见问答
  - **相关产品**：同品类下最多 3 个相关产品
  - **CTA 按钮**：引导至询盘中心（预填该产品）
- **SEO**：动态生成 title/description（基于产品 SEO 字段）

#### 3.1.4 询盘中心（RFQ）

- **路径**：`/rfq?product=X`
- **数据来源**：`getRfqPageData(selectedProductSlug?)`
- **表单字段**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 目标产品 | 下拉选择 | 否 | 从已发布产品中选择 |
| 联系人姓名 | 文本 | 是 | — |
| 公司名称 | 文本 | 否 | — |
| 商务邮箱 | 邮箱 | 是 | 格式校验 |
| 电话/WhatsApp | 文本 | 否 | — |
| 目的地市场 | 文本 | 是 | 进口国家/地区 |
| 目标数量 | 数字 | 否 | — |
| 目标单价 | 数字 | 否 | — |
| 货币 | 下拉 | 否 | USD / EUR / CNY / SGD / AED |
| 需求说明 | 多行文本 | 是 | 详细描述采购需求 |

- **提交流程**：
  1. 表单验证（必填字段 + 邮箱格式）
  2. 根据产品关联供应商，若无则分配默认供应商
  3. 生成询盘编号（格式：`INQ-YYYYMMDD-XXXXXX`）
  4. 创建 Inquiry 数据库记录
  5. 关联买家用户（如已登录）
  6. 跳转到成功页面展示参考编号

- **错误处理**：missing-fields / invalid-email / no-supplier 三种错误码

#### 3.1.5 买家工作台

- **路径**：`/account`
- **登录要求**：需要 BUYER 角色登录
- **数据来源**：`getBuyerInquiries(userId)`
- **内容**：
  - 买家欢迎信息和用户名
  - "我的询盘"列表：按时间倒序展示，每项含状态标签、产品名称、需求描述、目的地、数量
  - 空状态：无询盘时引导浏览产品目录
  - 询盘详情链接（跳转到 `[locale]/account/inquiries/[id]`）

#### 3.1.6 询盘详情页

- **路径**：`/account/inquiries/:id`
- **登录要求**：需要 BUYER 角色登录
- **数据来源**：`getInquiryDetails(inquiryNumber)`
- **内容**：
  - 询盘基本信息：编号、状态标签、创建时间
  - 关联产品信息（名称、图片、分类）
  - 联系信息：姓名、邮箱、电话、公司、目的地
  - 需求详情：目标数量、目标价格、详细需求描述
  - 分配供应商信息
  - **供应商报价列表**：每条报价显示金额、最小起订量、交期、有效期、备注
  - 无报价空状态提示

#### 3.1.7 登录页

- **路径**：`/login`
- **角色限制**：仅 BUYER 角色
- **功能**：
  - 邮箱 + 密码凭据登录
  - 登录成功跳转 `/account`
  - 登录失败回显错误提示
  - 已登录用户显示跳转引导

#### 3.1.8 关于页

- **路径**：`/about`
- **内容**：平台定位说明、供应商筛选机制、买家工作流说明、响应承诺

## 4. 核心业务工作流

### 4.1 询盘驱动采购流程（Inquiry-driven Procurement）

```
买家浏览产品目录 → 进入产品详情 → 发起询盘
                                      │
                                      v
                           填写结构化 RFQ 表单
                           ├── 选择目标产品（可选）
                           ├── 填写联系信息与需求
                           └── 提交（游客也可提交）
                                      │
                                      v
                           系统创建 Inquiry 记录
                           ├── 自动生成询盘编号
                           ├── 关联供应商（根据产品）
                           ├── 关联买家用户（如已登录）
                           └── 状态：NEW
                                      │
                    ┌─────────────────┼─────────────────┐
                    v                 v                   v
              管理员后台查看      供应商工作台查看      买家工作台追踪
              所有询盘            分配给自己的询盘       自己的询盘历史
                                      │
                                      v
                           供应商创建 Quote（报价）
                           ├── 金额、货币、最小起订量
                           ├── 交期、有效期、备注
                           └── 报价编号
                                      │
                                      v
                           报价状态：DRAFT → SENT → ACCEPTED / REJECTED / EXPIRED
                           询盘状态：NEW → IN_REVIEW → QUOTED → NEGOTIATING
                                                               │
                                              ┌────────────────┼────────────────┐
                                              v                                 v
                                        CLOSED_WON                        CLOSED_LOST / EXPIRED
                                        （达成交易）                        （流失/过期）
                                              │
                                              v
                                       创建 Order → 支付 → 履约
```

### 4.2 直采流程（Direct Purchase）

适用于已公布参考定价和标准商业包的产品，买家可直接查看价格并进入订单流程。当前平台绝大多数产品仍通过询盘中心发起，直采模式作为补充路线。

### 4.3 实体状态机

| 实体 | 状态枚举 | 流转说明 |
|------|---------|---------|
| **User** | ACTIVE → PENDING / SUSPENDED / DISABLED | 管理员控制用户账号启用/停用 |
| **Supplier** | PENDING → APPROVED / REJECTED → SUSPENDED | 供应商申请后由管理员审核 |
| **Product** | DRAFT → PENDING_REVIEW → APPROVED / REJECTED → PUBLISHED → OFFLINE | 商品从草稿到上架到下线 |
| **Inquiry** | NEW → IN_REVIEW → QUOTED → NEGOTIATING → CLOSED_WON / CLOSED_LOST / EXPIRED | 从新询盘到成交或流失 |
| **Quote** | DRAFT → SENT → ACCEPTED / REJECTED / EXPIRED | 报价从草稿到发送到确认或拒绝 |
| **Order** | PENDING → CONFIRMED → PROCESSING → SHIPPED → COMPLETED / CANCELLED | 订单从待处理到完成或取消 |
| **Payment** | PENDING → AUTHORIZED → PAID / FAILED → REFUNDED / VOIDED | 支付从待处理到完成或退款 |

## 5. 数据模型概览

### 5.1 用户与认证

| 模型 | 说明 |
|------|------|
| **User** | 用户账号（id, email, name, role, status, passwordHash, lastLoginAt） |
| **Account** | Auth.js 第三方账号关联（OAuth provider 预留） |
| **Session** | Auth.js 数据库会话（当前使用 JWT 策略，此表较少依赖） |
| **VerificationToken** | 邮箱验证令牌 |

### 5.2 组织与供应商

| 模型 | 说明 |
|------|------|
| **Organization** | 组织（SUPPLIER / BUYER / INTERNAL 三种类型，含基本信息） |
| **OrganizationMember** | 组织成员关联（OWNER / STAFF 角色） |
| **Supplier** | 供应商档案（含营业执照、税号、认证状态、审核信息） |

### 5.3 产品目录

| 模型 | 说明 |
|------|------|
| **ProductCategory** | 产品分类（支持父子层级、SEO 字段、排序） |
| **Product** | 产品（交易模式、状态、价格区间、规格 JSON、SEO、软删除） |
| **ProductVariant** | 产品变体（SKU、价格、库存、选项值 JSON、重量） |
| **ProductImage** | 产品图片（MAIN / DETAIL / SCENE 三种类型） |

### 5.4 询盘与报价

| 模型 | 说明 |
|------|------|
| **Inquiry** | 买家询盘（客户信息、需求详情、附件 JSON、分配供应商） |
| **Quote** | 供应商报价（金额、最小起订量、交期、有效期、备注） |

### 5.5 订单与支付

| 模型 | 说明 |
|------|------|
| **Order** | 订单（金额分解、收货地址 JSON、客户信息） |
| **OrderItem** | 订单项（**产品快照**：名称/SKU/图片/单价/数量，防止源数据变更影响历史订单） |
| **Payment** | 支付记录（支持 QUOTE / STRIPE / AIRWALLEX 三种 provider） |

### 5.6 内容与 AI

| 模型 | 说明 |
|------|------|
| **CmsPage** | CMS 页面（slug + locale 联合唯一，支持多语言） |
| **FaqItem** | FAQ 条目（可关联 CMS 页面 / 分类 / 产品） |
| **AiLog** | AI 调用审计日志（用途、模型、Token 用量、延迟、状态） |

### 5.7 命名约定

- 数据库表名和字段名使用 **snake_case**（通过 Prisma 的 `@map()` / `@@map()` 映射）
- 关联实体的 onDelete 策略：**Restrict** 保护核心数据 / **Cascade** 清理子实体 / **SetNull** 保留引用但允许删除关联
- 金额字段统一使用 `Decimal(12, 2)`
- 所有主键使用 UUID

## 6. 技术架构概览

### 6.1 技术栈

| 层级 | 技术选型 |
|------|---------|
| 前端框架 | Next.js 16 + React 19（App Router） |
| 构建工具 | Turborepo + pnpm workspace |
| 数据库 | PostgreSQL (Neon) + Prisma ORM |
| 认证 | Auth.js v5（NextAuth） |
| 国际化 | next-intl（en / zh） |
| AI 集成 | OpenAI SDK |
| 代码质量 | ESLint + Prettier + Husky + lint-staged |
| 部署 | Vercel |

### 6.2 工程结构

```
nongyechuhai/
├── apps/
│   ├── web/          @nongyechuhai/web      公共商店前端（买家端，端口 ~4000）
│   ├── admin/        @nongyechuhai/admin    后台管理系统（端口 ~4100）
│   └── supplier/     @nongyechuhai/supplier 供应商工作台（端口 ~4200）
├── packages/
│   ├── db/           @nongyechuhai/db        Prisma client + schema + seed
│   ├── auth/         @nongyechuhai/auth      Auth.js 封装 + 角色校验
│   ├── config/       @nongyechuhai/config    Zod 环境变量 + 共享常量
│   ├── ai/           @nongyechuhai/ai        OpenAI 客户端
│   └── ui/           @nongyechuhai/ui        共享 UI 组件 + 导航配置
└── tools/            开发辅助脚本（端口自动分配）
```

### 6.3 架构设计要点

**认证架构**：每个应用通过独立的 `auth.ts` 调用 `createAppAuth({ allowedRoles })`，分别限制为 BUYER / ADMIN / SUPPLIER。共享 auth 包封装 Auth.js v5 + JWT 会话 + bcrypt 密码验证。

**数据访问模式**：非传统 REST API，Server Components 直连数据库。使用 Prisma 类型安全的 select 对象定义查询，结果映射到应用层 DTO 类型。React `cache()` 用于请求周期内数据去重。

**路由架构**：
- web：`[locale]` 动态路由组 + next-intl 中间件处理语言
- admin + supplier：路由组布局在渲染前校验会话和角色

**渲染策略**：
- web：ISR（Incremental Static Regeneration），每 5 分钟重新验证
- admin + supplier：`force-dynamic`，始终获取实时数据

**样式方案**：纯 CSS（无框架），遵循 BEM 命名约定（如 `.site-header__brand`）。

## 7. 国际化支持

### 7.1 实现方式

- 框架：`next-intl`
- 支持语言：英语（en）、中文（zh）
- 路由策略：`localePrefix: 'always'`，URL 始终包含语言前缀（如 `/en/products`、`/zh/products`）
- 中间件：`apps/web/src/middleware.ts` 自动处理语言检测和重定向
- 消息文件：`apps/web/messages/en.json` / `apps/web/messages/zh.json`

### 7.2 覆盖范围

- **web 应用前台**：完整支持双语（首页、产品目录、产品详情、询盘、登录、买家工作台）
- **admin 后台**：以中文为主，部分 UI 标签为英文
- **supplier 工作台**：以中文为主
- **CMS 内容**：`CmsPage` 和 `FaqItem` 通过 `locale` 字段支持多语言内容创建
